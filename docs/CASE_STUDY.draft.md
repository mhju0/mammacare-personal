# 케이스 스터디 (초안) — MammaCare: 이유식 알레르기 안전 추적

> ⚠️ **초안.** 면접·포트폴리오용 talking point 모음이다. 사실관계는 코드/문서를 1차 진실로 보고, 빈칸(📌)은 본인 경험·수치로 채운다.

---

## 1. 문제 (Problem)

이유식 초기에는 새로운 재료를 **하루 한 가지씩, 며칠에 걸쳐 관찰**하며 도입한다. 알레르기 반응은 섭취 직후가 아니라 **수 시간~사흘 뒤**에 나타날 수 있어서, "무엇을 언제 먹였고 어떤 반응이 있었는지"를 정확히 추적하는 것이 안전의 핵심이다.

- **사용자:** 이유식을 막 시작/진행 중인 영아의 보호자. 수면 부족·정신없는 상황에서 모바일로 짧게 자주 본다.
- **고통점:** 정보가 머리/메모/사진에 흩어진다. 재도입·교차반응·확정 알레르겐을 일관되게 관리하기 어렵다.
- **왜 중요한가:** 알레르기 안전은 이 도메인에서 **가장 심각도 높은 문제**다. 알레르겐 필터 실패는 Critical급 결함이다.

📌 _개인 동기/현장 관찰이 있으면 1–2문장 추가(설득력↑)._

---

## 2. 결정 (Decision)

### 2.1 범위를 좁혔다 — "종합 육아앱 → 알레르기 안전 도구"
- 원래 종합 맘마케어에서 **알레르기 안전 추적 하나**로 집중. 솔로 프로젝트에서 완결성과 깊이를 확보하기 위한 의도적 축소.
- 트레이드오프: 기능 범위 ↓, 대신 핵심 플로우의 신뢰성·디자인 완성도 ↑.

### 2.2 AI/클라우드 의존을 제거했다 (부활 금지)
- Azure OpenAI·Speech·Language·Blob, AI 챗봇/식단, STT, NLP, Content Safety 전부 제거.
- 이미지: 클라우드 → 로컬 `backend/uploads/` + 보호 엔드포인트 `/api/media`.
- **이유:** (a) 외부 키·비용 없이 **로컬에서 완결**되어 재현·시연이 쉽다, (b) 데모/포트폴리오에서 통제 가능한 결정론적 동작, (c) 개인정보(아기 건강 데이터)를 외부로 보내지 않음.
- 트레이드오프: "AI 추천"의 화려함 ↓. 대신 **규칙 기반의 설명 가능한 로직**으로 신뢰성 확보.

### 2.3 DB 변경을 Alembic이 아니라 수동 SQL로 관리
- `backend/manual_sql/`에 **검증형 SQL**(`BEGIN/COMMIT` + 적용 전/후 `SELECT`)로 구조 변경을 기록.
- **이유:** `create_all()`은 신규 테이블만 만들고 기존 컬럼 타입/제약/enum을 자동 변경하지 않는다. 이 한계를 숨기지 않고, 변경마다 **사람이 pre/post 상태를 눈으로 검증**하고 커밋하는 절차로 안전성을 담보.
- 트레이드오프: 자동 마이그레이션 편의 ↓. 대신 위험한 DDL을 의식적으로 다룸(롤백 경로 명시).

### 2.4 "재테스트를 막던" 잔재 제약을 제거 — 단, 더 중요한 불변식은 보존
- 발견: `uq_ingredient_testing_baby_ingredient`(UNIQUE(baby_id, ingredient_id))가 **한 아기+한 재료를 전체 기간에 1회만** 허용 → 완료한 재료의 **재테스트를 영구히 차단**. ORM 모델에는 선언이 없는 **잔재** 제약이었다.
- 처리: [`001_drop_ingredient_testing_full_unique.sql`](../backend/manual_sql/001_drop_ingredient_testing_full_unique.sql)로 멱등 DROP(`IF EXISTS`). 1회 제거하면 `create_all()`이 다시 만들지 않음.
- **핵심:** 동시에 진행되는 테스트가 겹치면 안 된다는 **별개의 불변식**은 `ex_ingredient_testing_no_overlap`(EXCLUDE 제약)이 강제한다. 이 파일은 그것을 **건드리지 않는다**. → "재테스트 허용"과 "동시 1개" 두 규칙을 분리해 각자 올바른 메커니즘에 맡김.

## 3. 설계 (Design)

### 3.1 신호등 디자인 시스템 — "색이 곧 기능"
- 상태 = 신호등: **안전(초록)/테스트중(노랑)/반응(빨강)** + 신뢰·액션(Clinic 블루).
- 색을 토큰화(`theme.css`의 `:root` + Tailwind `@theme` 매핑): `safe-bg/fg`, `testing-bg/fg`, `reaction-bg/fg`.
- 접근성: **색만으로 의미 전달 금지** → `StatusChip`이 아이콘+텍스트(안전/테스트중/반응)를 함께 표시. `role="status"`, `aria-label` 부여.
- 베이스: Clinic(의료 신뢰) + 자연 한 스푼(세이지). 지친 부모 대상 → 정보 밀도 낮추고 여백 충분히.

### 3.2 대시보드를 히어로로
- 로그인 + 활성 아기 → 홈이 **알레르기 대시보드**(`HomeRoute.tsx`): 신호등 요약 → 다음 도입 추천(월령 기반 규칙) → 진행 중 테스트 관찰 진행바 → 최근 기록.
- 빈/로딩/에러 상태를 **happy path와 동등하게** 설계(스켈레톤, 재시도 버튼, 차분한 한국어 빈 상태 문구).

### 3.3 도메인 규칙을 식별자 기준으로
- 알레르기 비교는 **이름 문자열이 아니라 `ingredient_id`** 로. 동음이의/표기 흔들림에 의한 오탐·누락 방지.
- 테스트 생성은 사용자가 고른 재료 기준(수동). 같은 아기가 완료한 재료를 **다시 테스트할 수 있어야** 한다(§2.4).

### 3.4 백엔드 경계
- 모든 라우터 `/api` 아래(`/api/v1` 미사용). 현재 사용자 `/api/users/me`.
- Auth: JWT **access token만**(RefreshToken 미사용). OAuth는 fragment(`#`) 토큰 전달 + HMAC state.
- 코드 규약: async only, `httpx` only, `logging` only, 라우터 얇게, 에러 메시지 한국어, 본인 리소스 아니면 404.

## 4. 결과 (Result)

- **로컬에서 완결되는** 풀스택 앱: 외부 클라우드 키 없이 가입→아기→재료→알레르기 테스트→리포트까지 동작.
- 알레르기 추적의 **핵심 불변식을 제약으로 표현**: 재테스트 허용(잔재 UNIQUE 제거) + 동시 1개(EXCLUDE 보존)를 분리.
- **일관된 디자인 시스템**(토큰 → 컴포넌트 → 화면)으로 상태를 색으로 전달, 접근성 병행.

📌 _정량 지표가 있으면 추가:_ 화면 수, 엔드포인트 수, 빌드 시간, Lighthouse, 번들 크기 등.

### 잘한 점 (talking points)
- 위험한 변경(DB 제약)을 **검증 절차로 감싸** 안전하게 처리한 경험.
- "더 많은 기능"이 아니라 **범위 축소로 완성도**를 높인 제품 판단.
- 외부 의존 제거로 **재현 가능·프라이버시 친화** 데모 확보.

### 한계 / 다음 단계 (정직하게)
- 추천은 현재 **규칙 기반**(월령 필터). 다음: 영양 균형·교차반응까지 고려한 백엔드 추천 엔드포인트로 이관(코드 내 `TODO(M3)`).
- 대시보드에서 **활성 아기가 없을 때**의 빈 상태 처리 보강 필요(아기 등록 유도). _(야간 작업 로그 PHASE 3 참고)_
- 자동화 테스트 커버리지 📌, CI 파이프라인 📌.

## 부록 — 예상 질문 & 답변 뼈대

- **왜 Alembic을 안 썼나?** → `create_all()` 한계를 인지하고, 위험 DDL을 pre/post 검증으로 명시적으로 다루기 위해. 팀/규모가 커지면 Alembic 도입이 맞다(트레이드오프 인지).
- **왜 AI를 뺐나?** → 비용/키 없는 로컬 완결, 결정론적 시연, 건강 데이터 프라이버시. 기능보다 신뢰.
- **재테스트 버그를 어떻게 찾았나?** → 모델에 없는 잔재 제약이 도메인 규칙("재테스트 가능")과 충돌. 제약을 도메인 불변식 단위로 분리(UNIQUE 제거 / EXCLUDE 유지).
- **접근성은?** → 색만으로 상태 전달 금지, 아이콘+텍스트+`aria-label`, 충분한 대비.
